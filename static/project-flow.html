面试怎么讲

为什么用 Agent？问题类型不固定，需要动态决定查知识库、当前时间、日志还是监控。
为什么先宽召回再重排？向量检索保证召回率，Cross-Encoder 提升排序精度，兼顾效果和上下文成本。
多轮对话怎么做？MemorySaver 以 session_id 作为 thread_id 保存 LangGraph checkpoint。
一句话Agent 负责决策，Milvus 负责召回，FlashRank 负责精排，Qwen 负责生成。

面试怎么讲

为什么要 Replan？故障诊断无法一次规划准确，需要根据日志和指标结果动态调整。
怎么避免无限循环？限制最多 8 个执行步骤，执行 5 步后禁止再次扩展计划。
MCP 的价值？把日志和监控封装成标准工具，Agent 与具体数据源解耦。
一句话规划拆任务，执行拿证据，复盘调路径，最后出报告。

模块级拆解

按“职责、输入输出、关键实现、设计理由、深挖回答”理解代码，而不是只背技术名词。

应用入口与生命周期

app/main.py
职责创建 FastAPI、注册四类路由、挂载静态文件、配置 CORS，并管理 Milvus 生命周期。
关键点lifespan 启动时 connect，关闭时 close；STATIC_DIR 使用绝对路径，避免 cwd 改变导致资源丢失。
取舍应用启动依赖 Milvus 成功，更适合强依赖知识库的原型；生产可改为健康降级或延迟连接。
面试回答“入口层只做装配和生命周期管理，业务逻辑下沉到 service，避免路由变成胖控制器。”

API 与 SSE 适配层

app/api/chat.py · aiops.py · file.py
职责校验请求、调用服务、把内部事件转换为 HTTP JSON 或 SSE message。
事件聊天输出 content/done/error；AIOps 输出 plan/step_complete/report/complete/error。
价值浏览器能边接收边渲染，降低长耗时 LLM 和多步诊断的体感延迟，也让过程可观察。
深挖点“SSE 是单向服务端推送，当前场景不需要 WebSocket 的双向长连接，协议更简单。”

RAG Agent 编排

app/services/rag_agent_service.py
职责创建 ChatQwen Agent，合并本地工具与 MCP 工具，提供同步和流式查询、会话读取与清理。
状态MemorySaver 使用 session_id 作为 thread_id，保存 LangGraph checkpoint，实现多轮上下文隔离。
策略系统提示约束“内部知识库优先，知识不足再联网”，减少外部信息覆盖内部规范。
注意代码定义了 trim_messages_middleware，但当前 create_agent 未注册它；不能声称历史修剪已经生效。

知识检索工具

app/tools/knowledge_tool.py
输入输出输入 query；输出格式化上下文和原始 Document artifact，工具声明为 content_and_artifact。
流程Milvus fetch_k 宽召回 → 可选 FlashRank 重排 → 截断 rag_top_k → 带来源和标题格式化。
失败捕获异常并返回错误文本与空文档，Agent 可据此诚实降级，而不是中断整个对话。
面试回答“把检索封装成 Tool 后，是否检索由 Agent 决策，RAG 不再是每个问题都固定执行的流水线。”

向量召回与重排

vector_store_manager.py · rerank_service.py
召回默认从 Milvus 获取 20 个候选，向量相似度适合快速缩小语义搜索空间。
重排FlashRank Cross-Encoder 同时看 query 与 passage，精排后保留 3 条，并写入 _rerank_score。
降级模型延迟初始化；重排异常或映射不完整时退化为向量原顺序 Top 3。
取舍“两阶段检索用少量本地精排计算换取更高相关性，并控制最终 Prompt 长度。”

文档切分

app/services/document_splitter_service.py
Markdown先按 h1/h2/h3 标题切分并保留层级元数据，再用 RecursiveCharacterTextSplitter 二次切分。
参数chunk_max_size=800、overlap=100；小于 300 字符的相邻块会尝试合并。
理由标题切分保留结构语义，递归切分控制模型上下文粒度，overlap 避免边界信息断裂。
深挖点参数不是永恒最优，应通过 Recall@K、回答正确率和 token 成本做数据驱动调优。

Embedding 与索引

vector_embedding_service.py · vector_index_service.py
EmbeddingDashScope text-embedding-v4，统一输出 1024 维；文档批量 embed，查询单条 embed。
幂等索引前按 metadata._source 删除同源旧分片，然后重新切分和批量写入。
IDMilvus auto_id=False，每个 Document 使用 UUID 主键。
注意上传成功但索引失败时接口仍返回 200，目前只记录日志；生产应返回 indexing_status 或异步任务 ID。

Milvus 连接与 Schema

app/core/milvus_client.py
Schemabiz Collection 包含 id、1024 维 vector、content、JSON metadata，默认 2 shards。
启动检查 Collection，不存在则创建；存在但向量维度不匹配时会删除并重建。

AIOps 状态图

app/services/aiops_service.py · state.py
状态input、plan、past_steps、response；past_steps 用 operator.add 追加，保留完整证据链。

面试回答“显式状态图让每个阶段可测试、可追踪，也比一个超长 ReAct 循环更容易限制成本和终止条件。”

Planner / Executor / Replanner

app/agent/aiops/*.py
Planner检索内部经验、获取工具描述，通过 Pydantic 结构化输出 Plan.steps。
Executor只执行 plan[0]；LLM bind_tools 选工具，ToolNode 执行，再把工具结果交给 LLM 总结。
Replanner结构化输出 continue/replan/respond；最多 8 步，执行 5 步后禁止 replan。
取舍单步执行便于每轮根据证据纠偏；代价是 LLM 调用次数增加，需要步数上限控制延迟和费用。

MCP 客户端与工具服务

app/agent/mcp_client.py · mcp_servers/
客户端MultiServerMCPClient 延迟单例初始化，同时接入 cls:8003 与 monitor:8004。
可靠性Interceptor 最多重试 3 次，等待 1s、2s；最终返回 isError 工具结果而非抛出。
工具CLS 提供时间、地域、主题和日志查询；Monitor 提供 CPU、内存时序和阈值判断。
价值“MCP 把外部能力标准化，Agent 只依赖工具 schema；替换 CLS 或 Prometheus 时不必重写编排。”

本地工具与可选联网搜索

高频追问与可复述回答

先用 20 秒短答建立结论，再根据面试官兴趣展开实现与取舍。

问题分类
1. 请你用一分钟介绍这个项目

可直接回答：这是一个面向实验室双机视频传输场景的智能知识与运维平台。我把分散的实验文档做成 Milvus RAG 知识库，用 LangGraph Agent 支持多轮问答；故障诊断侧采用 Plan-Execute-Replan，把日志、CPU、内存等能力通过 MCP 接成标准工具，诊断过程通过 SSE 实时返回。项目的重点不是单次调用大模型，而是把知识、工具、状态和可观察执行流程组合成完整系统。

可能追问：为什么不是普通聊天机器人？回答业务闭环：文档入库、检索问答、日志指标联动诊断。

2. 为什么选择 FastAPI + LangGraph，而不是全写在一个服务函数里？

短答：FastAPI 负责协议和依赖装配，LangGraph 负责有状态、可循环、可中断的 Agent 工作流，两者职责不同。

展开：普通 RAG 查询相对线性，但 AIOps 会根据中间证据决定继续、改计划或结束。用状态图能显式定义节点、状态字段和条件边，每个节点可独立测试，SSE 也能直接映射节点更新。如果写成一个大函数，状态、重试、终止和观测会互相缠绕。

3. RAG 的完整链路是什么？

可直接回答：文档侧先校验落盘，按 Markdown 标题和递归字符切分，调用 text-embedding-v4 生成 1024 维向量，写入 Milvus。查询侧 Agent 决定调用 retrieve_knowledge，先从 Milvus 宽召回 20 条，再用 FlashRank Cross-Encoder 重排到 3 条，最后把带来源的上下文交给 Qwen 生成答案。

补充：重排失败会退化为向量原顺序截断，保证服务可用。

4. 为什么要“Top 20 召回 + Top 3 重排”？

短答：向量检索快但排序精度有限，Cross-Encoder 精度高但计算贵，所以先宽召回缩小集合，再精排。

深入：Embedding 是分别编码 query 和 document，适合大规模 ANN；Cross-Encoder 联合编码 query-passage，更能识别细粒度语义。20 和 3 是当前配置，不应当拍脑袋固定，应该用 Recall@20、MRR、答案正确率、延迟和 token 成本做联合调参。

5. chunk=800、overlap=100 是怎么定的？

回答原则：不要说这是最佳值。

当前文档以运维说明和 Markdown 为主，800 字符能保留一个相对完整的检查步骤，100 字符 overlap 降低关键信息跨边界被切断的概率。项目还先按标题层级切分，并合并过小块。更严谨的做法是建立问答评测集，对不同 chunk 大小、重叠和 Top K 做网格实验。

6. 如何避免模型编造答案？

第一，系统提示要求知识库优先且不得编造工具未返回内容；第二，检索结果带来源和标题，便于引用；第三，工具失败返回显式错误，最终报告要求如实写明失败；第四，AIOps 报告基于 past_steps 证据生成。当前仍不是强保证，生产可增加引用校验、无证据拒答、结构化 claim-evidence 对齐和事实评测。

7. 多轮对话是怎么实现的？会有什么问题？

LangGraph 使用 MemorySaver 作为 checkpointer，API 传入的 session_id 映射为 thread_id，因此不同会话的消息状态隔离。优点是实现简单、适合演示；缺点是进程重启丢失、无法多实例共享、历史增长会占内存。代码虽然定义了保留系统消息和最近对话的 trim 函数，但当前未挂到 create_agent，所以我会把它作为待完善项，而不会声称已生效。

生产方案：Postgres/Redis checkpointer + TTL + 摘要记忆 + 用户级鉴权。

8. Plan-Execute-Replan 与 ReAct 有什么区别？

ReAct 通常是模型在一个循环里交替思考和调用工具，灵活但过程容易发散。Plan-Execute-Replan 先显式生成多步计划，每次只执行一步，再根据累计证据评估继续、改计划或结束。它更适合长链路诊断，因为计划、历史和终止条件都是显式状态，代价是 LLM 调用次数更多。

9. AIOps 状态如何更新？为什么 past_steps 用 operator.add？

状态包含 input、plan、past_steps、response。Executor 返回 plan[1:] 来覆盖剩余计划，同时返回一个新的 (task, result)；past_steps 标注 operator.add 后，LangGraph 会追加而不是覆盖历史。这样 Replanner 和最终报告能看到完整执行证据，也方便前端展示进度。

10. 如何防止 Agent 无限循环或成本失控？

代码有三层限制：Replanner 优先级是 respond 高于 continue 高于 replan；执行达到 5 步后禁止再次 replan；达到 8 步强制生成最终响应。另外 replan 的新步骤不能多于当前剩余步骤。生产还会增加总 token、总耗时、工具调用次数、单工具超时和用户取消信号。

11. MCP 在项目里解决了什么问题？

MCP 把日志和监控系统暴露为带 schema 的标准工具。Agent 通过 MultiServerMCPClient 动态获取工具，不需要知道 CLS 或监控服务的 SDK 细节。这样工具服务可独立部署和替换，编排层只依赖统一协议。当前连接两个服务：CLS 日志工具和 Monitor CPU/内存工具。

追问安全：生产要增加工具白名单、参数校验、租户隔离、只读权限和审计日志。

12. MCP 工具失败怎么办？

客户端拦截器最多调用 3 次，重试间隔是 1 秒和 2 秒的指数退避。全部失败后不向上抛异常，而是返回 isError=True 的 CallToolResult。这样 Executor 会把失败作为一步证据写入 past_steps，最终报告可以明确说明数据缺失，而不是整条诊断链崩溃。

可改进：加入 jitter、熔断、按错误类型判断是否重试，以及全链路 deadline。

13. 日志与监控如何共同定位视频传输故障？

先用监控判断服务端或客户端是否存在 CPU、内存压力，再按告警时间窗口查询两端日志，寻找编码失败、连接拒绝、超时或缓冲异常。Replanner 根据证据决定是否补查另一个节点或缩小时间范围。最终报告要把“现象、指标、日志、根因、建议”串起来，而不是只罗列工具返回。

14. 为什么用 SSE，不用 WebSocket？

这里主要是客户端发起一次 HTTP 请求，服务端持续返回 token 和诊断状态，通信方向基本是单向的。SSE 基于 HTTP，浏览器处理简单，断线语义和代理兼容性也更直接。若未来需要用户中途交互式修改计划、双向控制或高频二进制数据，再考虑 WebSocket。

15. 系统有哪些降级设计？

Tavily 没有 API Key 时根本不注册；FlashRank 初始化或重排失败就用向量原顺序；MCP 失败经重试后返回错误结果；Executor 异常会跳过当前步骤并记录失败；知识工具异常返回空 artifact。降级目标是保住主流程，但要通过日志和最终报告暴露降级，不能静默伪装成功。

16. 这个项目距离生产环境还缺什么？

主要有五类：持久化会话与多实例状态共享；真实 CLS/Prometheus 数据源；鉴权、租户隔离和工具权限；异步索引任务与状态反馈；完整可观测和评测，包括 trace、token、延迟、Recall@K、答案正确率。还要处理 Milvus Schema 迁移，不能维度不匹配就直接删库。

17. 如何评价 RAG 和 Agent 的效果？

离线层建立问题、标准答案、相关文档和预期工具轨迹数据集；检索看 Recall@K、MRR、NDCG，生成看事实正确率、引用一致性和拒答准确率，Agent 看任务成功率、工具选择正确率、平均步数和循环率。在线层看 P95 延迟、首 token 时间、错误率、token 成本和用户反馈。

18. 如果并发量上来，瓶颈在哪里？怎么优化？

瓶颈主要是外部 LLM/Embedding 调用、FlashRank CPU、Milvus 检索和长连接数量。可以把上传索引改为队列异步任务；Embedding 批处理与缓存；FlashRank 做进程池或独立服务；限制每用户并发与 Agent 步数；SSE 服务水平扩展；会话 checkpoint 外置；工具调用设置超时、限流和熔断。

19. 为什么代码里既有 LLMFactory，又直接使用 ChatQwen？

这是项目演进留下的两种适配方式。LLMFactory 基于 ChatOpenAI，便于切换任何 OpenAI-compatible 提供商；当前 Agent 主链使用 langchain_qwq.ChatQwen，是为了更直接适配通义工具调用和流式行为。面试时我会如实说明当前没有完全统一，下一步应让 Agent 依赖抽象模型工厂，并针对结构化输出、工具调用和流式能力做 provider capability 配置。

20. 为什么仓库里有两套向量搜索实现？

VectorSearchService 是较底层的 pymilvus 搜索路径，直接生成 query vector 并用 L2 检索；当前 RAG 主链已经演进为 VectorStoreManager，通过 langchain_milvus Retriever 宽召回，再接 FlashRank 重排。两者不是同时运行。工程上应该保留统一 Retriever 抽象后删除 legacy 路径，避免搜索参数和距离语义不一致。

项目边界、风险与生产化演进

面试中主动区分“当前已实现”和“下一步设计”，可信度通常比堆砌技术名词更重要。

当前定位：这是一个完整可演示的智能运维原型，核心工作流、RAG、MCP、SSE 已经打通；日志和监控 MCP 当前主要生成演示数据，MemorySaver 也是进程内状态。回答时应明确原型边界，再说明如何接入真实基础设施。

当前边界与技术债

  • MemorySaver 进程重启后会话丢失，不支持多实例共享。
  • 消息裁剪函数已经定义，但尚未注册到 Agent，中长会话可能持续增长。
  • CLS 与 Monitor MCP 当前为演示实现，监控数据包含动态模拟逻辑。
  • 上传接口在索引失败时仍返回成功，调用方无法准确得知知识库状态。
  • Milvus 维度变化会直接删除旧 Collection，缺少无损迁移。
  • CORS 允许任意来源，接口缺少鉴权、限流和租户隔离。

生产化改造顺序

  • P0：接入真实日志/Prometheus，只读凭证和工具白名单。
  • P0：会话改为持久化 checkpointer，增加 TTL、用户鉴权和数据隔离。
  • P1：上传改异步任务，返回 task_id，支持索引进度、失败重试和回滚。
  • P1：增加超时、熔断、限流、全链路 deadline 与取消机制。
  • P1：建立离线评测集和 OpenTelemetry trace，记录 prompt、工具、token 与延迟。
  • P2：Collection 版本化、灰度重建与 alias 原子切换。

关键设计取舍

  • Agent 动态工具选择更灵活，但确定性和测试难度高于固定 RAG 链。
  • Plan-Execute-Replan 更可控，但比单轮 ReAct 产生更多 LLM 调用。
  • 两阶段检索提升精度，但增加本地重排延迟和模型资源。
  • SSE 适合单向过程输出，但不适合复杂双向实时控制。
  • 同步自动索引简单直观，但大文件会阻塞请求，应迁移到任务队列。
  • 工具错误作为证据继续执行提高可用性，但需要防止错误被模型误解为正常数据。

应该补齐的指标

  • 检索：Recall@K、MRR、NDCG、重排增益、无结果率。
  • 生成：事实正确率、引用命中率、无依据回答率、拒答准确率。
  • Agent：任务成功率、工具选择准确率、平均步骤数、replan 率、强制终止率。
  • 性能：首 token、端到端 P50/P95、工具延迟、Milvus 延迟、并发 SSE 数。
  • 成本：每次会话 token、每次诊断 LLM 调用数、Embedding 与重排资源。
  • 稳定性：MCP 重试率、降级率、索引失败率、会话恢复率。
